The Cardinal Rule: On-Chain vs Off-Chain Data
Blockchain storage is expensive, permanent, and public. Every byte stored on-chain costs real gas and is visible to every node on the network forever. This has direct consequences for cost, privacy, and legal compliance.
Store ON-CHAIN when data...
Keep OFF-CHAIN when data...
Is required by smart contract logic (e.g. bid amount, deadline)
Is a human-readable description, title, or image
Must be verified or enforced without trusting any server
Can be verified by storing only its keccak256 hash on-chain
Is a financial balance or token ownership record
Contains personal details (name, email, ID number, address)
Is an access control record (who is a verifier, who is an owner)
Is subject to GDPR right-to-erasure — immutability prevents deletion
Is a cryptographic commitment (a hash, not the secret itself)
Is large in size — images, PDFs, JSON metadata, documents
The IPFS pattern (mandatory for content data):  Store your file or JSON on IPFS, get back a CID (a 46-character content hash). Store ONLY that CID on-chain. Anyone can verify the content by fetching from IPFS and hashing it — the on-chain CID is the proof of authenticity.
Gas cost at a glance:  SSTORE (new slot) = 20,000 gas. Each non-zero calldata byte = 16 gas. A 500-character description stored as a string on-chain is approximately 40,000 gas. The same description on IPFS with only a 32-byte CID on-chain is approximately 512 gas. That is a 78x reduction.
GDPR and immutability:  Blockchain data is permanent. GDPR Article 17 gives users the right to have their personal data erased. These two facts are irreconcilable if personal data is stored on-chain. The solution is always: store personal data off-chain (encrypted, deletable), store only a non-reversible hash on-chain.
